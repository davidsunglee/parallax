# Performance methodology is bound but numeric targets are deferred

TypeScript binds to the shared `M13` performance methodology now and defers the numeric targets until a real implementation can produce a baseline. The methodology is the durable, comparable part of `M13`; concrete `p50` / `p95` and memory ceilings invented against a non-existent implementation would be fabricated, not measured, so they wait for a first baseline run.

What is bound for V1: TypeScript uses the shared benchmark fixtures under `core/compatibility/benchmarks/` (`read-mix.yaml`, `deep-fetch.yaml`, `milestone-write.yaml`), the nearest-rank percentile for `wallTimeMs.p50` / `wallTimeMs.p95`, the M13 report shape fixed in `core/spec/m13-performance.md` (`generatedAt`, `dialect`, `benchmarks[].{fixture, model, datasetRows, workloads[]}`, `memory.{peakBytes, steadyBytes}`), and the `parallax-conformance benchmark` command as the execution hook. The adapter command returns that report shape under `report` in the standard conformance envelope; writing the same object to `report.json` is only an artifact-copy concern. Running the same fixtures against the same deterministically generated datasets at the same scale keeps a TypeScript number directly comparable to the Python reference figures and to future runs.

What is deferred: the absolute numeric `p50` / `p95` wall-time targets per workload family and the peak / steady memory ceilings. These are recorded as deferred placeholders with an explicit tracking note — once the first TypeScript implementation runs the benchmark suite and records a baseline, the spec is updated with grading targets derived from that baseline. Setting numeric ceilings now was rejected as premature and unfounded.

What changed (slice adoption): the **conformance claim** no longer includes `M13`. Earlier drafts of §4.5 claimed `m13` in `modules` and the `benchmark` command in `commands`, treating benchmark execution as inside the V1 claim with only the numeric targets deferred. TypeScript V1 now **is** the canonical `slice-mvp-1` Conformance Slice (TS-0064), and that slice does not claim benchmarks: `m13` and the `benchmark` command are dropped from the §1.1 `describe` claim. This ADR's substance is unchanged — TypeScript still binds the shared M13 methodology and report shape now and still defers the numeric targets — but the *first build does not claim benchmark execution in its conformance slice*. The benchmark surface (the `benchmark` command and the `m13` module claim) lands after the first slice, at which point this section's numeric targets are set from the first baseline. The `expectRoundTrips` invariant below remains binding for V1 regardless, because it is enforced through the M12 round-trip-count layer that the slice does claim, not only through the deferred benchmark runner.

The `expectRoundTrips` invariant is explicitly carved out of the deferral and affirmed as a hard, non-placeholder requirement: a deep fetch costs `1 + levels` round trips regardless of fan-out, never N+1. The TypeScript adapter MUST honor it. It is enforced in both directions the core already enforces it — `M13`'s benchmark runner compares actual round trips to a workload's `expectRoundTrips`, and `M12`'s round-trip-count layer asserts `len(goldenSql[dialect]) == roundTrips` for every case. This invariant is binding for V1 even though the wall-time and memory numbers are not.
