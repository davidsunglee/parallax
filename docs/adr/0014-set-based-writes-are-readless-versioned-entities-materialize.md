# Set-based writes are readless; versioned entities materialize

A set-based update or delete targeting a predicate or an unresolved result collection executes readless — one set-based statement, no object materialization — for non-versioned entities. An optimistic-locking (versioned) entity is the exception: the write materializes. The runtime resolves the predicate to rows through the observing read — recording each row's framework-owned observed version, and in locking mode taking the shared in-transaction read lock (`m-read-lock`) — then issues one keyed per-object UPDATE per resolved row, gated on the observed version in optimistic mode and ungated version-advancing in locking mode. There is no set-based versioned UPDATE template, because the optimistic gate binds a per-row observed version that a single set-based statement cannot carry; materializing rides every row's version advance and gate on the framework-owned observed value (ADR 0013), turning a set-based write that would otherwise be an undetectable lost update into per-object conflict-detectable writes. Read-before-write thereby becomes universal for versioned entities — a keyed update requires a prior observe, a set-based update performs it — matching Reladomo's operation-based list setters.
