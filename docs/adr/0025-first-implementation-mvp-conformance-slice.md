# `first-implementation-mvp` is a named Conformance Slice, not a module tier

The first agent-buildable build of Parallax is encoded as a named **Conformance Slice / profile** called `first-implementation-mvp`, distinct from the MVP / fast-follow / definitely-do **module tiers** in `scope-and-tiers.md`. A Conformance Slice is, per `CONTEXT.md`, "a declared subset of the compatibility corpus that an implementation claims through the conformance adapter for a specific implementation milestone" — it is explicitly *not* a module tier. The tiers describe *parity scope* at whole-module granularity (what a complete implementation must eventually cover); the slice describes what one early build *honestly claims right now*, and it may defer *parts* of a module without redefining that module's boundary.

This distinction matters because the **MVP tier** and the **`first-implementation-mvp` slice** share the word "MVP" but are different things. The MVP tier claims whole modules — including `M8` *with* its identity cache and query cache, and PK-generation strategies. The slice claims a strictly smaller surface: Postgres only; `M8`'s transaction / unit-of-work behavior but **not** its caches; **no** PK generation, aggregation, value objects, inheritance, M9 detach, error classification, bitemporal rectangle-split writes, MariaDB, M13 benchmarks, or M14 coherence. Reading "build the MVP" as "build the MVP tier" silently means "implement full M8 caches and PK generation," which is exactly the confusion this slice removes.

The slice is **include-driven**: membership is a single per-case tag, `first-implementation-mvp`, appended to the ~97 included cases (family-selected: all of `00xx` / `02xx` / `03xx` / `05xx`, plus `0603` / `0604` / `0607` / `0612` / `0613`, `0703` / `0704` / `0707` / `0708`, and `0801`–`0805`). A case is *in the slice* precisely when it carries that tag and also passes the broad module / dialect / shape filters of the canonical `describe` claim. Nothing is selected by the *absence* of a tag, so the slice is immune to the corpus's known tag hazards (the `mariadb` literal tag on only one MariaDB-golden case; the single-case `identity cache` tag). The canonical claim is checked into `scope-and-tiers.md` as a fenced JSON block and is the single source of truth.

Core schemas and JSON+YAML serde conformance are **unchanged** — the slice is additive tags plus documentation plus a consistency gate. It carries **no `profile` wire key**: the slice name lives only in documentation and tags, never in a `describe` document, because `describeOk` is `additionalProperties: false` at the top level and inside `capabilities`. A new consistency gate, `profile_errors()` behind `dep_graph_check --profile`, parses the canonical claim and asserts the tagged set can never silently drift from it (every claimed module has a tagged case; every tagged case's shape and module tags fall inside the claim; every tagged case has a Postgres golden). This sits beside the existing parity coverage gate, which — like the tiers — is left untouched.

**Alternatives rejected.** Modeling the first build as a *module tier* was rejected because it would force whole-module semantics and could not defer cache/identity behavior without splitting `M8`. Modeling it as full module parity was rejected as far larger than a first build. A new profile-manifest file format (with its own schema and harness tooling) was rejected as unnecessary machinery for a fixed 97-case slice when an additive tag plus the existing `caseTags.include` seam already encodes membership. Adding a `profile` field to the conformance-adapter schema was rejected to keep core schemas unchanged; the include tag already *is* the machine-readable slice marker.
