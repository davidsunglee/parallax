# Optimistic locking composes over temporal entities via the processing-from column

A processing-axis temporal entity carries no version column, so its optimistic-lock key is DERIVED from the processing-from column: the observed processing-from (`in_z`) value is the version analogue (Reladomo's `IN_Z` rule). In optimistic mode a milestone close/inactivate `UPDATE` gains an `and in_z = ?` gate bound to the `in_z` the unit of work observed, so a concurrent chain that left a fresh current row (a new `in_z`) makes a stale close match zero rows — the conflict signal — rather than silently re-closing it (a lost update the current-row predicate alone would miss). A zero-row close is an error in ANY mode (never silent): a retriable optimistic conflict in optimistic mode, a distinct non-retriable stale/consistency error in locking mode. On success no version numbers exist to bump — the gate rides only on the close `UPDATE`s the write already issues (one per closed/inactivated current row, each binding that row's observed `in_z`), and the chained replacement rows are plain ungated `INSERT`s whose fresh `in_z = txInstant` is the advance. Combining an explicit `optimisticLocking` attribute with `asOfAttributes` on one entity is invalid, and a business-temporal-only entity cannot participate in optimistic mode (it has no processing axis to derive the key from). M10 composes over M7 (the new `M10 --> M7` dependency edge); M10 owns the conflict/retry contract and M7 owns the write shapes. This builds on ADR 0029 (framework-owned version values) and ADR 0031 (bounded automatic retry).
