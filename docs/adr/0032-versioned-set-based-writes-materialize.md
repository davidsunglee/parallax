# Versioned set-based writes materialize

A set-based update or delete targeting an optimistic-locking (versioned) entity materializes: the runtime resolves the predicate to rows through the observing read — recording each row's framework-owned observed version, and in locking mode taking the M8 shared lock — then issues one keyed per-object UPDATE per resolved row, gated on the observed version in optimistic mode and ungated version-advancing in locking mode. There is no set-based versioned UPDATE template, because the optimistic gate binds a per-row observed version that a single set-based statement cannot carry. This supersedes ADR 0011 in part: its readless set-based execution stands for non-versioned entities, but a versioned entity materializes so every row's version advance and gate ride the framework-owned observed value (ADR 0029), turning a set-based write that would otherwise be an undetectable lost update into per-object conflict-detectable writes. Read-before-write thereby becomes universal for versioned entities — a keyed update requires a prior observe, a set-based update performs it — matching Reladomo's operation-based list setters.
