# Slice tags follow the slice-naming convention

A Conformance Slice's name *is* its `caseTags.include` tag, and slice tags follow a fixed convention: lowercase, matching `^slice-[a-z0-9][a-z0-9-]*$` — a `slice-` prefix (which makes slice tags structurally recognizable in a case's flat `tags` array, the way `^m\d+$` marks module tags) followed by a short, language-neutral purpose name and an ordinal. Under this convention the canonical first slice is renamed `first-implementation-mvp` → `slice-mvp-1` across the corpus, docs, tooling, and the TypeScript packages; the ordinal leaves room for successors (`slice-mvp-2`) and siblings (`slice-temporal-1`) without any name going stale. Slice membership, the `--profile` / `--coverage` gates, and the 99-case claim are unchanged — only the tag literal moved.

**Considered options.** Per-language slice tags (e.g. `slice-typescript-1`) were rejected: slices are core-defined and language-neutral, so per-language tags would force re-tagging the same 99 shared corpus files for every adopter, and *which* language claims a slice already lives in the `adapter` identity of that language's `describe` response — not in the tag. The long-form `slice-first-implementation-mvp` was also rejected: "first-implementation" goes stale the moment a second language adopts the slice, while the ordinal already carries the sequencing the name needs.

Historical ADRs — `docs/adr/0025-first-implementation-mvp-conformance-slice.md` and the TypeScript `0064-adopt-first-implementation-mvp-slice.md` — retain the old tag name as a record of the original decision; this ADR records the rename.
