# Prisma Transactions and No ORM Session

## Summary

Prisma Client exposes transaction control as query APIs, not as an ORM
`Session`, persistence context, or identity-map scope. Its documented
transaction surface is nested writes, bulk operations, sequential
`$transaction([])`, and interactive `$transaction(async tx => ...)`, with
documented isolation-level and timeout options where supported by the connector.
The docs describe Prisma Client itself as an auto-generated type-safe query
builder, and the Prisma repository README states that Prisma Client queries
return plain JavaScript objects, not managed model instances
([Prisma Client intro](https://www.prisma.io/docs/orm/prisma-client/setup-and-configuration/introduction),
[README lines 32-38](https://github.com/prisma/prisma/blob/9acd8dcdc9395df8f8c4de241b8cfbfd2da227e9/README.md#L32-L38),
[README lines 249-268](https://github.com/prisma/prisma/blob/9acd8dcdc9395df8f8c4de241b8cfbfd2da227e9/README.md#L249-L268)).

Primary-source inference: because the reviewed Prisma docs/source expose
`PrismaClient`, generated input/output types, explicit CRUD methods, and
transaction-bound client proxies, but no `Session`/persistence-context/identity
map API, Prisma should be treated as transaction-only prior art for this slice.
SQLAlchemy/Hibernate are used here only as labels for the session/persistence
context pattern; no non-Prisma source was used.

## Transaction API Surface

Prisma documents a database transaction as read/write operations that succeed or
fail as a whole under ACID properties, and lists three main techniques:
dependent writes use nested writes, independent writes use `$transaction([])` or
batch operations, and read-modify-write logic uses interactive transactions
([transactions overview](https://www.prisma.io/docs/orm/prisma-client/queries/transactions#transactions-and-batch-queries)).

Nested writes are relation-aware write calls that perform multiple operations on
related records in one transaction, such as creating a user and related posts in
one `create` call. The transaction guide says nested writes should be used when
later operations need IDs generated by earlier operations, and that Prisma rolls
back the entire nested-write transaction if any operation fails
([nested writes](https://www.prisma.io/docs/orm/prisma-client/queries/transactions#nested-writes),
[dependent writes](https://www.prisma.io/docs/orm/prisma-client/queries/transactions#dependent-writes)).

Batch operations are documented as transactional for `createMany()` /
`createManyAndReturn()`, `updateMany()` / `updateManyAndReturn()`, and
`deleteMany()`. Prisma separately documents that `updateMany` and `deleteMany`
do not support nested writes, so bulk mutation and relation-tree mutation are
separate API shapes
([batch operations](https://www.prisma.io/docs/orm/prisma-client/queries/transactions#batch-operations),
[nested writes with bulk operations](https://www.prisma.io/docs/orm/prisma-client/queries/transactions#can-i-use-nested-writes-with-bulk-operations)).

Sequential `$transaction([])` takes an array of Prisma Client queries, executes
them in array order, and treats them as one atomic operation for independent
writes. The docs call out that this form cannot pass generated IDs between
operations; if operation B needs an ID produced by operation A, use nested writes
or precomputed IDs
([sequential operations](https://www.prisma.io/docs/orm/prisma-client/queries/transactions#sequential-operations),
[`$transaction([])` API](https://www.prisma.io/docs/orm/prisma-client/queries/transactions#transaction-api),
[precomputed IDs](https://www.prisma.io/docs/orm/prisma-client/queries/transactions#scenario-pre-computed-ids-and-the-transaction-api)).
The client runtime implements this form by requiring every array element to be
an unawaited `PrismaPromise`, then attaching batch transaction metadata
including index, isolation level, `maxWait`, and `timeout`
([getPrismaClient.ts lines 773-803](https://github.com/prisma/prisma/blob/9acd8dcdc9395df8f8c4de241b8cfbfd2da227e9/packages/client/src/runtime/getPrismaClient.ts#L773-L803)).

Interactive transactions take an async callback. Prisma passes that callback a
transaction-scoped Prisma Client instance, and any Prisma Client call invoked on
that `tx` object is encapsulated in the transaction. The docs say Prisma commits
when the callback reaches the end and automatically rolls back if the callback
throws
([interactive transactions](https://www.prisma.io/docs/orm/prisma-client/queries/transactions#interactive-transactions)).
The runtime starts an engine transaction, executes the callback with a proxied
client, commits on success, and rolls back while rethrowing the original error
on failure
([getPrismaClient.ts lines 812-887](https://github.com/prisma/prisma/blob/9acd8dcdc9395df8f8c4de241b8cfbfd2da227e9/packages/client/src/runtime/getPrismaClient.ts#L812-L887),
[getPrismaClient.ts lines 899-960](https://github.com/prisma/prisma/blob/9acd8dcdc9395df8f8c4de241b8cfbfd2da227e9/packages/client/src/runtime/getPrismaClient.ts#L899-L960)).

Isolation levels are configurable for sequential and interactive transactions
with the `isolationLevel` option. Prisma documents support for
`ReadUncommitted`, `ReadCommitted`, `RepeatableRead`, `Snapshot`, and
`Serializable` when the underlying database supports them; MongoDB has no
isolation-level support in this API. Prisma uses the database-configured default
isolation level unless overridden for a transaction
([transaction isolation](https://www.prisma.io/docs/orm/prisma-client/queries/transactions#transaction-isolation-level)).

Timeouts are documented for interactive transactions: `maxWait` is the maximum
wait to acquire a transaction and defaults to 2000 ms, while `timeout` is the
maximum transaction runtime and defaults to 5000 ms. The runtime also stores
constructor-level default transaction options with `maxWait`, `timeout`, and
`isolationLevel`, and applies those defaults to both batch and interactive
transaction metadata
([interactive transaction options](https://www.prisma.io/docs/orm/prisma-client/queries/transactions#interactive-transactions),
[getPrismaClient.ts lines 82-87](https://github.com/prisma/prisma/blob/9acd8dcdc9395df8f8c4de241b8cfbfd2da227e9/packages/client/src/runtime/getPrismaClient.ts#L82-L87),
[getPrismaClient.ts lines 489-493](https://github.com/prisma/prisma/blob/9acd8dcdc9395df8f8c4de241b8cfbfd2da227e9/packages/client/src/runtime/getPrismaClient.ts#L489-L493)).

For write conflicts or deadlocks, Prisma documents `P2034` as the error code
returned when a transaction fails due to write conflict/deadlock, and shows
retrying the transaction in application code after setting `Serializable`
isolation where appropriate
([transaction timing issues](https://www.prisma.io/docs/orm/prisma-client/queries/transactions#transaction-timing-issues)).

## Session/Persistence-Context Absence or Closest Equivalents

Prisma's closest long-lived client-side object is `PrismaClient`, not a
session. The docs describe Prisma Client as an auto-generated type-safe query
builder tailored to the schema, and recommend generally creating one
`PrismaClient` instance because multiple instances create multiple connection
pools
([Prisma Client intro](https://www.prisma.io/docs/orm/prisma-client/setup-and-configuration/introduction),
[database connections](https://www.prisma.io/docs/orm/prisma-client/setup-and-configuration/databases-connections#long-running-processes)).
That singleton guidance is connection-pool lifecycle guidance, not a statement
that `PrismaClient` is an identity map.

The data model provides the foundation for generated Prisma Client queries, and
generated types are TypeScript input/output utilities used with methods such as
`prisma.user.create(...)`, `prisma.user.update(...)`, `select`, and `include`.
The type-safety docs describe result types as derived from a model, arguments,
and operation, not as instrumented model classes
([README lines 157-175](https://github.com/prisma/prisma/blob/9acd8dcdc9395df8f8c4de241b8cfbfd2da227e9/README.md#L157-L175),
[generated types](https://www.prisma.io/docs/orm/prisma-client/type-safety#what-are-generated-types),
[type utilities](https://www.prisma.io/docs/orm/prisma-client/type-safety#type-utilities)).

CRUD operations are explicit method calls. Reads use methods such as
`findUnique`, `findMany`, and `findFirst`; writes use `create`, `update`,
`updateMany`, `upsert`, `delete`, and `deleteMany` with explicit `where` and
`data` objects. This is materially different from a unit-of-work session that
tracks mutations to loaded objects and flushes dirty state later
([CRUD overview](https://www.prisma.io/docs/orm/prisma-client/queries/crud),
[update examples](https://www.prisma.io/docs/orm/prisma-client/queries/crud#update)).

The transaction-scoped `tx` object in an interactive transaction is also not a
session/persistence context. It is a proxied Prisma Client whose requests carry
transaction metadata, and the runtime removes transaction-denied client
properties from that proxy. Its scope is the callback's database transaction,
not an object identity or dirty-checking scope
([interactive transactions](https://www.prisma.io/docs/orm/prisma-client/queries/transactions#interactive-transactions),
[getPrismaClient.ts lines 899-920](https://github.com/prisma/prisma/blob/9acd8dcdc9395df8f8c4de241b8cfbfd2da227e9/packages/client/src/runtime/getPrismaClient.ts#L899-L920)).

Object state tracking is application-controlled. The transaction docs' optimistic
concurrency example adds a `version` field, reads it into application memory,
and then includes that version in an `updateMany` predicate while incrementing
it. The docs present this as building optimistic concurrency control yourself,
not as a built-in persistence-context version check
([optimistic concurrency control](https://www.prisma.io/docs/orm/prisma-client/queries/transactions#optimistic-concurrency-control)).

## Cache/Identity Semantics

There is no documented Prisma guarantee that two reads of the same row within a
client, request, or transaction return the same JavaScript object identity. The
repo README states that Prisma Client queries return plain JavaScript objects,
and the docs/source reviewed do not expose a session-local identity map
([README lines 249-268](https://github.com/prisma/prisma/blob/9acd8dcdc9395df8f8c4de241b8cfbfd2da227e9/README.md#L249-L268)).

There is also no documented unit-of-work dirty checking session. The documented
write surface requires explicit write calls with explicit data, and the runtime
transaction implementation carries request/transaction metadata rather than
tracking loaded object instances
([CRUD update](https://www.prisma.io/docs/orm/prisma-client/queries/crud#update),
[getPrismaClient.ts lines 973-980](https://github.com/prisma/prisma/blob/9acd8dcdc9395df8f8c4de241b8cfbfd2da227e9/packages/client/src/runtime/getPrismaClient.ts#L973-L980)).

Prisma does have query execution optimizations that can look cache-like but do
not provide identity-map semantics. The query optimization docs say Prisma's
dataloader automatically batches `findUnique()` queries in the same tick, and
later clarify that it groups `findUnique()` calls with the same parameters and
selection set into a single `findMany()` query. That is query batching for the
n+1 problem, not an object identity or state-tracking guarantee
([query optimization](https://www.prisma.io/docs/orm/prisma-client/queries/advanced/query-optimization-performance#solving-the-n1-problem)).

Prisma Accelerate is the documented result-cache feature. It is optional and
uses per-query `cacheStrategy` with `ttl`, `swr`, and optional tags for read
operations such as `findUnique`, `findMany`, `count`, `aggregate`, and `groupBy`.
The docs say `cacheStrategy` is not supported on write operations such as
`create`, and `withAccelerateInfo()` reports `cacheStatus: "none"` when no cache
strategy was specified and the database query was executed
([Accelerate cacheStrategy](https://www.prisma.io/docs/accelerate/reference/api-reference#cachestrategy),
[supported operations](https://www.prisma.io/docs/accelerate/reference/api-reference#supported-prisma-client-operations),
[Accelerate info](https://www.prisma.io/docs/accelerate/reference/api-reference#withaccelerateinfo)).
Accelerate known limitations also state that raw query responses cannot be
cached
([Accelerate limitations](https://www.prisma.io/docs/accelerate/more/known-limitations#cannot-cache-raw-queries)).

The Prisma Client runtime source also exposes `queryPlanCacheMaxSize`, described
as a maximum size for a query plan cache. That is an implementation/config cache
for query plans, not a cache of row objects or a persistence-context identity
map
([getPrismaClient.ts lines 125-139](https://github.com/prisma/prisma/blob/9acd8dcdc9395df8f8c4de241b8cfbfd2da227e9/packages/client/src/runtime/getPrismaClient.ts#L125-L139)).

## Failure Modes and Lifecycle Edge Cases

Interactive transactions are callback-scoped. The docs warn to keep them short:
long-running transactions hurt performance and can cause deadlocks, and Prisma
specifically cautions against network requests and slow queries inside
transaction functions
([interactive transactions](https://www.prisma.io/docs/orm/prisma-client/queries/transactions#interactive-transactions)).

A Prisma transaction is connection-bound. The transaction guide says that queries
inside a transaction run on the same connection, and a database connection can
execute only one query at a time; therefore, wrapping operations inside
`Promise.all()` within a transaction still executes them serially
([using `$transaction` within `Promise.all()`](https://www.prisma.io/docs/orm/prisma-client/queries/transactions#using-transaction-within-promiseall)).

Connection lifecycle is owned by Prisma Client and the driver/pool, not by a
user-visible ORM session. Prisma Client lazily connects on the first query or
when `$connect()` is called, creates a connection pool, and disconnects when the
Node.js process ends. `$disconnect()` closes all pool connections, and the
connection-pool docs state that individual database connections are not exposed
to developers when using driver adapters
([connection management](https://www.prisma.io/docs/orm/prisma-client/setup-and-configuration/databases-connections/connection-management),
[connection pool](https://www.prisma.io/docs/orm/prisma-client/setup-and-configuration/databases-connections/connection-pool#manual-database-connection-handling)).

Creating multiple `PrismaClient` instances creates multiple pools and can
exhaust database connection limits. Prisma recommends one shared client instance
for long-running applications and instantiating outside the function handler in
serverless code to improve reuse while the handler remains warm
([database connections](https://www.prisma.io/docs/orm/prisma-client/setup-and-configuration/databases-connections),
[connection management](https://www.prisma.io/docs/orm/prisma-client/setup-and-configuration/databases-connections/connection-management#disconnect)).

The runtime contains nested interactive-transaction scope checks: it prevents
concurrent sibling nested transactions, requires nested transactions to close in
reverse order, and rolls back to avoid leaking an open top-level transaction if
an ordering violation occurs. This is transaction-scope bookkeeping in the
source, not a documented ORM session identity map
([getPrismaClient.ts lines 819-887](https://github.com/prisma/prisma/blob/9acd8dcdc9395df8f8c4de241b8cfbfd2da227e9/packages/client/src/runtime/getPrismaClient.ts#L819-L887)).

## Implications for Parallax Session-Cache Slice

Prisma is a useful contrast case for a Parallax slice that intentionally has a
session cache but no process-wide cache or coherence concern: Prisma's default
ORM surface shows that transaction atomicity, connection pinning, and request
lifecycle can be delivered without object identity reuse, dirty checking, or a
session-local cache. A Parallax session-cache slice therefore needs a distinct
session/cache contract; a transaction callback alone would not imply identity
map behavior under the Prisma model
([interactive transactions](https://www.prisma.io/docs/orm/prisma-client/queries/transactions#interactive-transactions),
[README lines 249-268](https://github.com/prisma/prisma/blob/9acd8dcdc9395df8f8c4de241b8cfbfd2da227e9/README.md#L249-L268)).

Prisma's lifecycle split is also relevant: `PrismaClient` is recommended as a
shared process object for pool reuse, while the transaction is callback/query
scope and result objects are plain values. For Parallax, a session cache can be
scoped below any process-wide client/adapter object and above individual SQL
statements, without inheriting Prisma-style process-wide pooling concerns as
cache-coherence concerns
([database connections](https://www.prisma.io/docs/orm/prisma-client/setup-and-configuration/databases-connections#long-running-processes),
[connection pool manual handling](https://www.prisma.io/docs/orm/prisma-client/setup-and-configuration/databases-connections/connection-pool#manual-database-connection-handling)).

Prisma's explicit-update model highlights the semantic difference Parallax must
pin down if it supports session caching: cached identity can be read-through and
session-local without becoming a unit-of-work dirty checker. Prisma requires
explicit `update`/`updateMany` calls, and its OCC example requires an explicit
version predicate; Parallax can choose which of those behaviors it adopts, but
the session-cache slice should not accidentally infer dirty checking from
identity reuse
([CRUD update](https://www.prisma.io/docs/orm/prisma-client/queries/crud#update),
[optimistic concurrency control](https://www.prisma.io/docs/orm/prisma-client/queries/transactions#optimistic-concurrency-control)).

Prisma's caching story separates three concepts that Parallax should keep
separate in this slice: identity map/session cache, query batching, and result
caching. Prisma has documented dataloader batching and optional Accelerate read
result caching, but neither is a session-local identity map or unit-of-work
cache. This supports keeping Parallax's session cache local to the session and
out of any process-wide cache/coherence design for this slice
([query optimization](https://www.prisma.io/docs/orm/prisma-client/queries/advanced/query-optimization-performance#solving-the-n1-problem),
[Accelerate cacheStrategy](https://www.prisma.io/docs/accelerate/reference/api-reference#cachestrategy)).
