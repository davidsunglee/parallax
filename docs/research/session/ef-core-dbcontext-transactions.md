# Entity Framework Core DbContext Session-Like and Transaction Management

## Summary

EF Core's `DbContext` is the closest EF Core concept to a session: the public API documentation says its lifetime begins at construction, ends at disposal, and is designed for a single short unit of work; the EF Core source summary likewise describes a `DbContext` instance as a session with the database and a combination of Unit of Work and Repository patterns ([DbContext lifetime](https://learn.microsoft.com/en-us/ef/core/dbcontext-configuration/#the-dbcontext-lifetime), [DbContext source](https://github.com/dotnet/efcore/blob/abad2bec8c0bf2b4bb86e69951a1169413bb9ea4/src/EFCore/DbContext.cs#L13-L14)). Its responsibilities include querying, tracking entity instances, detecting changes, coordinating `SaveChanges`, and providing per-context identity resolution for tracked entities ([DbContext lifetime](https://learn.microsoft.com/en-us/ef/core/dbcontext-configuration/#the-dbcontext-lifetime), [change tracking overview](https://learn.microsoft.com/en-us/ef/core/change-tracking/), [tracking queries](https://learn.microsoft.com/en-us/ef/core/querying/tracking#tracking-queries), [identity resolution](https://learn.microsoft.com/en-us/ef/core/querying/tracking#identity-resolution)).

EF Core's transaction boundary is not the same thing as the `DbContext` lifetime. By default, one `SaveChanges` call is transactional when the database provider supports transactions; users can explicitly begin, commit, and roll back transactions through `DbContext.Database`; EF Core also documents savepoints, cross-context shared transactions, external `DbTransaction` integration, ambient `System.Transactions`, and retry/execution-strategy interactions ([transactions default behavior](https://learn.microsoft.com/en-us/ef/core/saving/transactions#default-transaction-behavior), [controlling transactions](https://learn.microsoft.com/en-us/ef/core/saving/transactions#controlling-transactions), [savepoints](https://learn.microsoft.com/en-us/ef/core/saving/transactions#savepoints), [cross-context transaction](https://learn.microsoft.com/en-us/ef/core/saving/transactions#cross-context-transaction), [external DbTransactions](https://learn.microsoft.com/en-us/ef/core/saving/transactions#using-external-dbtransactions-relational-databases-only), [execution strategies and transactions](https://learn.microsoft.com/en-us/ef/core/miscellaneous/connection-resiliency#execution-strategies-and-transactions)).

The cache guarantee is narrow: EF Core tracks and identity-resolves entities within a `DbContext`/query tracking scope, but no source reviewed states or implies a process-wide coherent entity cache. The docs instead repeatedly scope tracking to each context instance, document that no-tracking queries bypass the context change tracker and can reflect database state disregarding local changes, and show `StateManager`/`IdentityMap` internals owned by the context service path rather than a global object cache ([change tracking overview](https://learn.microsoft.com/en-us/ef/core/change-tracking/), [no-tracking queries](https://learn.microsoft.com/en-us/ef/core/querying/tracking#no-tracking-queries), [identity resolution](https://learn.microsoft.com/en-us/ef/core/querying/tracking#identity-resolution), [StateManager identity maps](https://github.com/dotnet/efcore/blob/abad2bec8c0bf2b4bb86e69951a1169413bb9ea4/src/EFCore/ChangeTracking/Internal/StateManager.cs#L455-L488), [IdentityMap dictionary](https://github.com/dotnet/efcore/blob/abad2bec8c0bf2b4bb86e69951a1169413bb9ea4/src/EFCore/ChangeTracking/Internal/IdentityMap.cs#L18-L36)).

## DbContext/session-like responsibilities

`DbContext` is short-lived by design. Microsoft documents the typical unit of work as creating a context, tracking entity instances returned from queries or added/attached to the context, changing those entities, calling `SaveChanges`/`SaveChangesAsync`, and disposing the context ([DbContext lifetime](https://learn.microsoft.com/en-us/ef/core/dbcontext-configuration/#the-dbcontext-lifetime)). The change-tracking docs repeat the same lifecycle and state that disposal is the normal way to stop tracking entities; they also note that explicit detaching or clearing is unnecessary in that normal lifecycle, though `ChangeTracker.Clear` is more efficient than detaching entities one by one when clearing is needed ([how to track entities](https://learn.microsoft.com/en-us/ef/core/change-tracking/#how-to-track-entities)).

Tracking is per `DbContext` instance. Entity instances become tracked when returned from database queries, explicitly attached by `Add`, `Attach`, `Update`, or similar methods, or detected as new entities connected to existing tracked entities; they stop being tracked when the context is disposed, the change tracker is cleared, or they are explicitly detached ([change tracking overview](https://learn.microsoft.com/en-us/ef/core/change-tracking/#how-to-track-entities)). The EF Core source exposes a `ChangeTracker` property whose summary says it provides access to information and operations for entity instances this context is tracking, and the property is initialized from the context's internal service provider ([DbContext.ChangeTracker source](https://github.com/dotnet/efcore/blob/abad2bec8c0bf2b4bb86e69951a1169413bb9ea4/src/EFCore/DbContext.cs#L151-L168)).

Tracked entities have one of EF Core's documented states: `Detached`, `Added`, `Unchanged`, `Modified`, or `Deleted`. `Added`, `Modified`, and `Deleted` drive insert/update/delete behavior on `SaveChanges`; `Unchanged` entities are tracked but do not generate database writes; `Detached` entities are not tracked by the context ([entity states](https://learn.microsoft.com/en-us/ef/core/change-tracking/#entity-states)). EF Core tracks changes at the property level, so a database update can include only modified properties for an entity in the `Modified` state ([entity states](https://learn.microsoft.com/en-us/ef/core/change-tracking/#entity-states)).

Query tracking is the default for queries returning entity types. In a tracking query, EF Core records entity instances in the change tracker; later detected changes are persisted during `SaveChanges`; EF Core also performs navigation fix-up between query results and entities already in the change tracker ([tracking queries](https://learn.microsoft.com/en-us/ef/core/querying/tracking#tracking-queries)). When materializing a tracking result, EF Core checks whether the entity is already tracked and returns the existing instance rather than overwriting current/original values with database values; if it is not already tracked, EF Core creates a new instance and attaches it to the context ([tracking queries](https://learn.microsoft.com/en-us/ef/core/querying/tracking#tracking-queries)).

No-tracking queries are documented for read-only scenarios and skip setup of change-tracking information. EF Core says no-tracking query results are based on the database and disregard local changes or added entities in the context; no-tracking can be selected per query with `AsNoTracking`, and the default query tracking behavior can be changed at the context level or configured via `UseQueryTrackingBehavior` ([no-tracking queries](https://learn.microsoft.com/en-us/ef/core/querying/tracking#no-tracking-queries), [configuring default tracking](https://learn.microsoft.com/en-us/ef/core/querying/tracking#configuring-the-default-tracking-behavior), [DbContextOptions common methods](https://learn.microsoft.com/en-us/ef/core/dbcontext-configuration/#other-dbcontext-configuration)).

Identity resolution is tied to tracking unless explicitly requested for a no-tracking query. A tracking query uses the change tracker for identity resolution and returns the same entity instance whenever the same entity appears multiple times in a result; ordinary no-tracking queries do not use the change tracker, do not perform identity resolution, and can return duplicate object instances for the same database row ([identity resolution in query tracking](https://learn.microsoft.com/en-us/ef/core/querying/tracking#identity-resolution)). `AsNoTrackingWithIdentityResolution` creates a stand-alone change tracker only for result materialization, so the query result has identity resolution but is not tracked by the context; after enumeration, that temporary tracker goes out of scope ([identity resolution in query tracking](https://learn.microsoft.com/en-us/ef/core/querying/tracking#identity-resolution)). The dedicated identity-resolution docs also state that no-tracking identity resolution requires keeping track of returned instances and that `AsNoTrackingWithIdentityResolution` keeps result instances unique without normal tracking ([identity resolution docs](https://learn.microsoft.com/en-us/ef/core/change-tracking/identity-resolution#identity-resolution-and-queries)).

Explicit tracking handles new and disconnected objects. `Add` places new entities in `Added` so `SaveChanges` inserts them; `Attach` places existing disconnected entities in `Unchanged`; `Update` behaves like `Attach` but marks entities `Modified`; and these operations apply to graphs of related entities, not just one root object ([explicit tracking introduction](https://learn.microsoft.com/en-us/ef/core/change-tracking/explicit-tracking#introduction), [inserting new entities](https://learn.microsoft.com/en-us/ef/core/change-tracking/explicit-tracking#inserting-new-entities), [attaching existing entities](https://learn.microsoft.com/en-us/ef/core/change-tracking/explicit-tracking#attaching-existing-entities), [updating existing entities](https://learn.microsoft.com/en-us/ef/core/change-tracking/explicit-tracking#updating-existing-entities)). Microsoft cautions that attaching entities to the same context they came from should not normally be needed, and routinely doing a no-tracking query and then attaching the returned entities is slower and harder to get right than using a tracking query ([explicit tracking introduction](https://learn.microsoft.com/en-us/ef/core/change-tracking/explicit-tracking#introduction), [identity resolution docs](https://learn.microsoft.com/en-us/ef/core/change-tracking/identity-resolution#identity-resolution-and-queries)).

Relationship loading participates in the same tracking surface. Eager loading uses `Include`/`ThenInclude` to include related data in query results, and EF Core automatically fixes up navigation properties to other entities already loaded into the context instance ([eager loading](https://learn.microsoft.com/en-us/ef/core/querying/related-data/eager#eager-loading)). Filtered include can be surprising for tracking queries because navigation fix-up can repopulate a filtered collection with previously tracked entities that do not match the filter; the docs suggest no-tracking or a fresh context for those cases, and a filtered navigation is considered loaded even when elements are missing from the filtered query ([filtered include caution](https://learn.microsoft.com/en-us/ef/core/querying/related-data/eager#filtered-include)). Explicit loading uses `DbContext.Entry(...).Collection(...).LoadAsync()` or `.Reference(...).LoadAsync()`, and separately loaded related entities are automatically wired to already loaded entities when change tracking is enabled ([explicit loading](https://learn.microsoft.com/en-us/ef/core/querying/related-data/explicit#explicit-loading)). Lazy loading can be enabled through proxies or `ILazyLoader`, is context-associated once the entity is attached, and can cause N+1 extra database round trips ([lazy loading with proxies](https://learn.microsoft.com/en-us/ef/core/querying/related-data/lazy#lazy-loading-with-proxies), [lazy loading without proxies](https://learn.microsoft.com/en-us/ef/core/querying/related-data/lazy#lazy-loading-without-proxies)).

`SaveChanges` is the change-tracker flush path. The docs describe the change-tracking approach as querying tracked data, mutating .NET properties, then calling `DbContext.SaveChanges()`, at which point EF Core detects changes by comparing loaded entities with snapshots and persists detected changes; `Add`/`Remove` similarly record tracked changes that are applied by `SaveChanges` ([saving data overview](https://learn.microsoft.com/en-us/ef/core/saving/#approach-1-change-tracking-and-savechanges)). The EF Core source for `SaveChanges`/`SaveChangesAsync` automatically calls `DetectChanges` when `AutoDetectChangesEnabled` is true before delegating to the `StateManager` save path ([DbContext SaveChanges source](https://github.com/dotnet/efcore/blob/abad2bec8c0bf2b4bb86e69951a1169413bb9ea4/src/EFCore/DbContext.cs#L609-L689), [DbContext SaveChangesAsync source](https://github.com/dotnet/efcore/blob/abad2bec8c0bf2b4bb86e69951a1169413bb9ea4/src/EFCore/DbContext.cs#L744-L810)).

## Transaction semantics

EF Core's default transaction behavior is per `SaveChanges` call. When a provider supports transactions, all changes in a single `SaveChanges` call are applied in a transaction; if any change fails, the transaction is rolled back and none of the changes are applied, giving all-or-nothing behavior for that call ([default transaction behavior](https://learn.microsoft.com/en-us/ef/core/saving/transactions#default-transaction-behavior)). The saving overview similarly states that, on databases that support it, `SaveChanges` automatically wraps multiple changes in a transaction ([saving data overview](https://learn.microsoft.com/en-us/ef/core/saving/#approach-1-change-tracking-and-savechanges)); the basic `SaveChanges` doc notes that most providers make `SaveChanges` transactional so operations are not left partially applied ([basic SaveChanges](https://learn.microsoft.com/en-us/ef/core/saving/basic#multiple-operations-in-a-single-savechanges)).

Explicit local transactions are created through the `DbContext.Database` API. Microsoft documents `BeginTransactionAsync`, `CommitAsync`, and rollback behavior around multiple `SaveChanges` calls and queries in one transaction; if the transaction is disposed without commit after failure, it is automatically rolled back in the sample's intended usage ([controlling transactions](https://learn.microsoft.com/en-us/ef/core/saving/transactions#controlling-transactions)). Transaction API behavior depends on providers: relational providers support transactions, while other provider types may throw or no-op when transaction APIs are called ([controlling transactions](https://learn.microsoft.com/en-us/ef/core/saving/transactions#controlling-transactions)).

Savepoints are used when `SaveChanges` runs inside an already active transaction. EF Core automatically creates a savepoint before saving data; if `SaveChanges` fails, it rolls back to that savepoint so the surrounding transaction remains as if the save attempt had not started, which can support correcting and retrying issues such as optimistic concurrency failures ([savepoints](https://learn.microsoft.com/en-us/ef/core/saving/transactions#savepoints)). Savepoints can also be manually created and rolled back via the transaction API; SQL Server MARS is a documented edge case where EF Core does not create savepoints and a failed `SaveChanges` can leave the transaction state unknown ([savepoints](https://learn.microsoft.com/en-us/ef/core/saving/transactions#savepoints)).

Cross-context transactions are documented but relational-only. Sharing a transaction across multiple context instances requires sharing both a `DbConnection` and a `DbTransaction`; the docs show externally constructing `DbContextOptions` with a shared connection and then using `UseTransactionAsync` for the second context ([cross-context transaction](https://learn.microsoft.com/en-us/ef/core/saving/transactions#cross-context-transaction), [share connection and transaction](https://learn.microsoft.com/en-us/ef/core/saving/transactions#share-connection-and-transaction)). The same transactions page documents sharing a transaction with non-EF ADO.NET code by starting a `DbTransaction` on a connection, running ADO.NET work, constructing a context with the same connection, and calling `UseTransactionAsync` ([external DbTransactions](https://learn.microsoft.com/en-us/ef/core/saving/transactions#using-external-dbtransactions-relational-databases-only)).

Ambient transactions are available through `System.Transactions` where the provider supports them. The docs show `TransactionScope` and explicit `CommittableTransaction`/`EnlistTransaction`, but also document limitations: provider support must be tested, distributed transactions require .NET 7.0+ on Windows, and `TransactionScope` lacks async commit/rollback support ([using System.Transactions](https://learn.microsoft.com/en-us/ef/core/saving/transactions#using-systemtransactions), [limitations of System.Transactions](https://learn.microsoft.com/en-us/ef/core/saving/transactions#limitations-of-systemtransactions)).

Execution strategies/retries constrain manual transaction control. EF Core's connection-resiliency docs say each query and each `SaveChangesAsync` is retried as its own unit when retries are enabled, but user-initiated transactions define a group of operations that must be replayed together; attempting this directly with a retrying strategy produces an exception instructing use of `DbContext.Database.CreateExecutionStrategy()` to execute the full transaction as a retriable unit ([execution strategies and transactions](https://learn.microsoft.com/en-us/ef/core/miscellaneous/connection-resiliency#execution-strategies-and-transactions)). The same docs highlight commit uncertainty: if a connection drops while a transaction is being committed, the resulting transaction state is unknown; one documented mitigation is to call `SaveChanges` with `acceptAllChangesOnSuccess: false`, verify success, and call `ChangeTracker.AcceptAllChanges()` later ([transaction commit failure](https://learn.microsoft.com/en-us/ef/core/miscellaneous/connection-resiliency#transaction-commit-failure-and-the-idempotency-issue), [state verification](https://learn.microsoft.com/en-us/ef/core/miscellaneous/connection-resiliency#option-3---add-state-verification)).

## Cache/identity semantics

The EF Core identity cache is a per-context tracking/identity-resolution mechanism, not a documented process-wide coherent cache. The docs phrase tracking as "each `DbContext` instance tracks changes made to entities" and state that tracked entities drive database changes on `SaveChanges`; entities are no longer tracked when the context is disposed, the change tracker is cleared, or they are detached ([change tracking overview](https://learn.microsoft.com/en-us/ef/core/change-tracking/), [how to track entities](https://learn.microsoft.com/en-us/ef/core/change-tracking/#how-to-track-entities)). The query docs say tracking queries check only whether an entity is already in the context, return the same instance if found, and do not overwrite current/original values from database values in that existing entry ([tracking queries](https://learn.microsoft.com/en-us/ef/core/querying/tracking#tracking-queries)).

Ordinary no-tracking queries make the boundary visible. They do not use the change tracker, do not perform identity resolution, return a new entity instance for duplicate occurrences of the same entity, and give results based on the database while disregarding local changes and added entities in the context ([no-tracking queries](https://learn.microsoft.com/en-us/ef/core/querying/tracking#no-tracking-queries), [identity resolution in query tracking](https://learn.microsoft.com/en-us/ef/core/querying/tracking#identity-resolution)). `AsNoTrackingWithIdentityResolution` uses a temporary stand-alone tracker for the query result only, and that tracker is not the context's tracker and goes out of scope after enumeration ([identity resolution in query tracking](https://learn.microsoft.com/en-us/ef/core/querying/tracking#identity-resolution)).

The source implementation matches the per-context identity-map model. `DbContext` exposes a unique `ContextId` for the context instance/pool lease, resolves dependencies including `IStateManager` through the context's internal services, and `SaveChanges` delegates to that state manager ([DbContext source](https://github.com/dotnet/efcore/blob/abad2bec8c0bf2b4bb86e69951a1169413bb9ea4/src/EFCore/DbContext.cs#L184-L192), [DbContext dependencies source](https://github.com/dotnet/efcore/blob/abad2bec8c0bf2b4bb86e69951a1169413bb9ea4/src/EFCore/DbContext.cs#L234-L242), [SaveChanges source](https://github.com/dotnet/efcore/blob/abad2bec8c0bf2b4bb86e69951a1169413bb9ea4/src/EFCore/DbContext.cs#L609-L689)). `StateManager` maintains identity maps keyed by EF Core keys, creates them lazily, and stores overflow maps in a dictionary; `IdentityMap<TKey>` stores a dictionary from key value to `InternalEntityEntry` ([StateManager identity maps](https://github.com/dotnet/efcore/blob/abad2bec8c0bf2b4bb86e69951a1169413bb9ea4/src/EFCore/ChangeTracking/Internal/StateManager.cs#L455-L488), [IdentityMap dictionary](https://github.com/dotnet/efcore/blob/abad2bec8c0bf2b4bb86e69951a1169413bb9ea4/src/EFCore/ChangeTracking/Internal/IdentityMap.cs#L18-L36)).

EF Core does have internal caches unrelated to entity-instance coherence. For example, `DbContextOptionsBuilder` exposes configuration for EF Core's internal service-provider caching and memory-cache integration, and the `DbContext` source caches `DbSet` instances by type/name within a context instance ([DbContextOptions common methods](https://learn.microsoft.com/en-us/ef/core/dbcontext-configuration/#other-dbcontext-configuration), [DbSet cache source](https://github.com/dotnet/efcore/blob/abad2bec8c0bf2b4bb86e69951a1169413bb9ea4/src/EFCore/DbContext.cs#L301-L343)). Those documented/internal caches should not be confused with an entity identity map: the docs' entity tracking and identity-resolution guarantees are scoped to the context or to the temporary stand-alone tracker used for `AsNoTrackingWithIdentityResolution` ([tracking queries](https://learn.microsoft.com/en-us/ef/core/querying/tracking#tracking-queries), [identity resolution in query tracking](https://learn.microsoft.com/en-us/ef/core/querying/tracking#identity-resolution)).

## Failure modes and lifecycle edge cases

`DbContext` is not thread-safe. Microsoft documents that EF Core does not support multiple parallel operations on the same context instance, including parallel async queries and explicit concurrent use from multiple threads; callers must await async calls immediately or use separate contexts for parallel operations ([avoiding threading issues](https://learn.microsoft.com/en-us/ef/core/dbcontext-configuration/#avoiding-dbcontext-threading-issues)). The docs also warn that an `InvalidOperationException` thrown by EF Core can put the context into an unrecoverable state, indicating a program error rather than a recoverable condition ([DbContext lifetime important notes](https://learn.microsoft.com/en-us/ef/core/dbcontext-configuration/#the-dbcontext-lifetime)).

Disposal is part of correctness, not just cleanup. The docs say disposing a context frees unmanaged resources and unregisters events/hooks to avoid memory leaks; `DbContextFactory`-created contexts are not managed by the application service provider and must be disposed by the application ([DbContext lifetime important notes](https://learn.microsoft.com/en-us/ef/core/dbcontext-configuration/#the-dbcontext-lifetime), [DbContext factory](https://learn.microsoft.com/en-us/ef/core/dbcontext-configuration/#use-a-dbcontext-factory)).

Tracking state can make queries return local/session-shaped results rather than a fresh database view. EF Core says no-tracking queries disregard local changes and added entities, which implies tracking queries may reflect tracked local state in identity resolution and navigation fix-up; the filtered include docs explicitly document that previously tracked entities can appear in filtered include results even when they fail the filter ([no-tracking queries](https://learn.microsoft.com/en-us/ef/core/querying/tracking#no-tracking-queries), [filtered include caution](https://learn.microsoft.com/en-us/ef/core/querying/related-data/eager#filtered-include)). The tracking docs also state that query results do not contain entities that were added to the context but not yet saved, limiting how far local tracking changes affect query result membership ([tracking queries](https://learn.microsoft.com/en-us/ef/core/querying/tracking#tracking-queries)).

Disconnected graphs require explicit state choices and can be easy to get wrong. Microsoft documents disconnected entities as objects queried by one context and later reattached to another, commonly after a web-client round trip; the application must reattach them with enough state information for `SaveChanges` to know what to insert or update ([explicit tracking introduction](https://learn.microsoft.com/en-us/ef/core/change-tracking/explicit-tracking#introduction)). Generated key values let EF Core infer new entities in a disconnected graph when keys are unset; with explicit keys, the application must set the right state ([generated versus explicit key values](https://learn.microsoft.com/en-us/ef/core/change-tracking/explicit-tracking#generated-versus-explicit-key-values), [attaching existing entities](https://learn.microsoft.com/en-us/ef/core/change-tracking/explicit-tracking#attaching-existing-entities)).

Transaction savepoints and retries have provider and connection-state edge cases. SQL Server MARS disables EF-created savepoints and can leave an active transaction in an unknown state after a failed `SaveChanges`; manual transactions are incompatible with implicitly invoked retrying execution strategies unless the full transaction is executed through the strategy delegate; and commit-time connection failure leaves transaction outcome unknown unless the application uses one of the documented mitigation patterns ([savepoints warning](https://learn.microsoft.com/en-us/ef/core/saving/transactions#savepoints), [execution strategies and transactions](https://learn.microsoft.com/en-us/ef/core/miscellaneous/connection-resiliency#execution-strategies-and-transactions), [transaction commit failure](https://learn.microsoft.com/en-us/ef/core/miscellaneous/connection-resiliency#transaction-commit-failure-and-the-idempotency-issue)).

Lazy loading is a loading convenience with runtime costs and hidden database access. The docs warn that lazy loading can cause extra round trips and the N+1 problem; it also relies on either proxy requirements (`virtual` navigations on inheritable classes) or injection of `ILazyLoader`/a delegate into entity types ([lazy loading with proxies](https://learn.microsoft.com/en-us/ef/core/querying/related-data/lazy#lazy-loading-with-proxies), [lazy loading without proxies](https://learn.microsoft.com/en-us/ef/core/querying/related-data/lazy#lazy-loading-without-proxies)).

## Implications for Parallax session-cache slice

For a Parallax slice that supports session caching but intentionally has no process-wide cache or coherence concerns, EF Core's useful design boundary is that the identity map belongs to the unit-of-work/session object, not to the process. EF Core's docs and source support a model where tracked identity resolution, change snapshots, state transitions, and navigation fix-up are local to one `DbContext` or to a temporary no-tracking identity-resolution query scope ([DbContext lifetime](https://learn.microsoft.com/en-us/ef/core/dbcontext-configuration/#the-dbcontext-lifetime), [tracking queries](https://learn.microsoft.com/en-us/ef/core/querying/tracking#tracking-queries), [identity resolution](https://learn.microsoft.com/en-us/ef/core/querying/tracking#identity-resolution), [StateManager identity maps](https://github.com/dotnet/efcore/blob/abad2bec8c0bf2b4bb86e69951a1169413bb9ea4/src/EFCore/ChangeTracking/Internal/StateManager.cs#L455-L488)).

The session-cache slice can separate identity caching from transaction ownership. EF Core allows the same `DbContext` to issue multiple implicit transactional `SaveChanges` calls across its lifetime, or to participate in an explicit transaction that spans multiple saves/queries; therefore the session lifetime and database transaction lifetime are related but not identical concepts ([default transaction behavior](https://learn.microsoft.com/en-us/ef/core/saving/transactions#default-transaction-behavior), [controlling transactions](https://learn.microsoft.com/en-us/ef/core/saving/transactions#controlling-transactions), [savepoints](https://learn.microsoft.com/en-us/ef/core/saving/transactions#savepoints)).

The cache contract can be narrow and explicit: a session may return the same in-memory object for the same key while tracked in that session, but read-only/no-track paths may bypass the identity map or use a temporary query-local identity-resolution structure. EF Core documents all three modes: tracking, no tracking, and no tracking with identity resolution ([tracking queries](https://learn.microsoft.com/en-us/ef/core/querying/tracking#tracking-queries), [no-tracking queries](https://learn.microsoft.com/en-us/ef/core/querying/tracking#no-tracking-queries), [identity resolution](https://learn.microsoft.com/en-us/ef/core/querying/tracking#identity-resolution)).

A minimal Parallax slice can treat process-wide coherence, invalidation, stale-object coordination across sessions, and second-level cache behavior as out of scope without contradicting the EF Core prior art reviewed here. EF Core's documented behavior is per-context tracking and identity resolution; the sources reviewed do not provide a process-wide entity identity cache guarantee, and no-tracking queries explicitly bypass context tracking ([change tracking overview](https://learn.microsoft.com/en-us/ef/core/change-tracking/), [no-tracking queries](https://learn.microsoft.com/en-us/ef/core/querying/tracking#no-tracking-queries), [identity resolution](https://learn.microsoft.com/en-us/ef/core/querying/tracking#identity-resolution)).

The slice should preserve visible lifecycle edges because EF Core exposes them as correctness boundaries: short-lived session lifecycle, dispose/clear/detach as tracking termination, non-thread-safe context usage, unrecoverable context state after certain EF exceptions, and explicit behavior for failed saves inside active transactions ([how to track entities](https://learn.microsoft.com/en-us/ef/core/change-tracking/#how-to-track-entities), [DbContext lifetime important notes](https://learn.microsoft.com/en-us/ef/core/dbcontext-configuration/#the-dbcontext-lifetime), [avoiding threading issues](https://learn.microsoft.com/en-us/ef/core/dbcontext-configuration/#avoiding-dbcontext-threading-issues), [savepoints](https://learn.microsoft.com/en-us/ef/core/saving/transactions#savepoints)).
