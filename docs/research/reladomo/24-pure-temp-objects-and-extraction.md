# Pure objects are cache-only (no-op persister), temp objects are real temp tables with scoped contexts, and extractor/reverse-engineering tools round-trip data and schema

> Part of [Research: Reladomo Core Features](00-index.md) — Reladomo @ commit

> repository (`../reladomo`). Path abbreviations: **`mithra/`** =
> `reladomo/src/main/java/com/gs/fw/common/mithra/`; **`genutil/`** =
> `reladomogenutil/src/main/java/com/gs/fw/common/mithra/generator/`.

**Pure objects.** Declared with root element `<MithraPureObject objectType="transactional">` (e.g. `reladomo/src/test/reladomo-xml/PureOrder.xml:20`) — same attributes/relationships/indices as regular objects, but no `DefaultTable`. Codegen swaps the persistence layer: `CoreMithraGenerator` renames the `DatabaseObject*` templates to `ObjectFactory*` for pure classes (`reladomogen/src/main/java/com/gs/fw/common/mithra/generator/CoreMithraGenerator.java:248-251`), producing `<Class>ObjectFactoryAbstract extends MithraAbstractPureObjectFactory` (`reladomogen/src/main/templates/transactional/ObjectFactoryAbstract.jsp:57`; runtime base at `mithra/portal/MithraAbstractPureObjectFactory.java:35`, implementing the 3-method `MithraPureObjectFactory` interface — `setFactoryParameter`/`loadFullCache`/`reloadFullCache`, `mithra/MithraPureObjectFactory.java:20-25`). The portal's persister is `new PureMithraObjectPersister(getFinderInstance())` (`reladomogen/.../MithraObjectTypeWrapper.java:2150-2152`): every write (`insert`, `update`, `delete`, `batchInsert`, …) is an empty method and every SQL read (`find`, `count`, `refresh`) throws `RuntimeException("not implemented")` (`mithra/portal/PureMithraObjectPersister.java:54-111,158-186`) — safe because runtime config forces `config.fullCache = true` and `isParticipatingInTx = true` ("pure objects can't be read-only") in `MithraConfigurationManager.lazyInitPureObjects` (`mithra/util/MithraConfigurationManager.java:721-782`, esp. 749-758), and `MithraAbstractObjectPortal` never bypasses the local cache when `isPureHome()` (`mithra/portal/MithraAbstractObjectPortal.java:624`). Runtime config is `<PureObjects notificationIdentifier="...">` with nested `MithraObjectConfiguration` entries (`reladomo/src/main/xsd/mithraruntime.xsd:31-45`; example `reladomo/src/test/resources/MithraConfigPartialCache.xml:371-373`). The required `notificationIdentifier` substitutes for a connection-manager-derived database identifier in cross-VM notification: the home instance appends its host address (`setPureHome`, `MithraAbstractObjectPortal.java:231-242`) and `MithraDatabaseIdentifierExtractor` uses `portal.getPureNotificationId()` as the subject (`mithra/notification/MithraDatabaseIdentifierExtractor.java:207`). Transactions work normally — writes enroll in the tx and commit/rollback against the cache only (see `PureMithraObjectPersister.enrollDatedObject`, 113-116).

**Temp objects.** Declared with `<MithraTempObject>` (`reladomo/src/test/reladomo-xml/OrderDriver.xml:20`) — pk attributes plus relationships that join to persistent objects. Their generated DatabaseObject extends `MithraAbstractTempObjectDatabaseObject[WithSource]` (`MithraObjectTypeWrapper.getDatabaseObjectSuperClass`, `reladomogen/.../MithraObjectTypeWrapper.java:2332-2343`), and the Finder gains `public static TemporaryContext createTemporaryContext(...)` (`reladomogen/src/main/templates/readonly/Finder.jsp:1499`). `TempContextContainer.createTemporaryContext` (`mithra/tempobject/TempContextContainer.java:119-139`) picks the context by ambient transaction: `MithraTxTempContext` (dropped via `TransactionLifeCycleListener` on commit/rollback, `mithra/tempobject/MithraTxTempContext.java:24,44`) or thread-local `MithraNonTxTempContext`; `init` creates the physical temp table immediately (`mithra/tempobject/AbstractMithraTemporaryContext.java:51-58`) and `destroy()` drops it. Separately, `TupleTempContext` (`mithra/tempobject/TupleTempContext.java:68`) is the internal mechanism behind large-IN queries: `SqlQuery` converts a set-based operation into a temp-table join when `setSize > databaseType.getUseTempTableThreshold()` (`mithra/finder/SqlQuery.java:751`; default `4*maxClauses`, `mithra/databasetype/AbstractDatabaseType.java:732-735`), inserting tuples via `MithraTuplePersister.insertTuples` (`TupleTempContext.java:240-266`) and dropping the table in `destroy()` (372-385); table creation/drop lives in `MithraAbstractDatabaseObject.createNonSharedTempTable`/`createSharedTempTable`/`dropTempTable`/`destroyTempContext` (`mithra/database/MithraAbstractDatabaseObject.java:610,698,755,2777`).

**Object-graph extraction** (`mithra/util/dbextractor/`). `MithraObjectGraphExtractor` "takes one or more root operations and extracts the data returned by those operations, and all related data to text files … parsed by MithraTestDataParser" (class javadoc, `MithraObjectGraphExtractor.java:67-79`). Constructor takes three strategy interfaces (92): `ExtractorConfig` (thread-pool size, per-relationship timeout, and an extract-threshold ratio that fails the extract when related/source object counts explode — `ExtractorConfig.java:25-48`), `MilestoneStrategy` (`addAsOfOperations(Operation)` adds as-of clauses so full/partial/no history is captured — `MilestoneStrategy.java:28-37`), and `OutputStrategy` (maps finder+source to output `File`). Shaping tools: `addRootOperation` (106, must not contain AsOfOperations), `addRelationshipFilter` (117, OR-merged), `ignoreRelationship` (127); traversal is multi-threaded with per-relationship timeouts. The simpler `DbExtractor` (`DbExtractor.java:65`) writes one class/operation at a time (`addClassToFile` 130, `addClassAndRelatedToFile` 135) in mithraTestData format — or columnar `.ccbf` when the filename ends so (201) — and can merge with existing file content.

**Reverse engineering and DDL** (`reladomogenutil`). `MithraObjectXmlGenerator` (`genutil/objectxmlgenerator/MithraObjectXmlGenerator.java:33`, an Ant `Task` with a `main()`) connects over JDBC, reads `DatabaseMetaData` tables/columns/PKs/FKs/indices (`processTableInfo`, 389-413), and emits one `<MithraObject>` XML per table (`generateMithraObjectXml`, 693-756): `DefaultTable`, `<Attribute>` per column with SQL→Java type mapping, `maxLength`, `primaryKey`, `identity` (795-822), `<Relationship>` per foreign key with inferred cardinality and join expression (758-774), and `<Index>` entries; milestone columns are recognized by the fixed name pairs `IN_Z/OUT_Z` (processingDate) and `FROM_Z/THRU_Z` (businessDate) and emitted as `<AsOfAttribute>` (`genutil/objectxmlgenerator/TableInfo.java:48-49`). Tables lacking a PK get a loud `NO PRIMARY KEY FOR THIS TABLE...` marker (731). The forward direction is `MithraDbDefinitionGenerator` (`genutil/dbgenerator/MithraDbDefinitionGenerator.java:27` wrapping `CoreMithraDbDefinitionGenerator`), which per object emits `.ddl` (CREATE TABLE), `.idx`, and `.fk` files through a dialect-specific `AbstractGeneratorDatabaseType` (`generateDdlFile/generateIdxFile/generateFkFile`, `genutil/dbgenerator/CoreMithraDbDefinitionGenerator.java:173-175`; dialects for H2, Sybase, Oracle, Postgres, Maria, UDB8.2 in the same package).

## Testing patterns

`TestPureTransactionalObject` (`reladomo/src/test/.../TestPureTransactionalObject.java`) exercises pure objects exactly like DB-backed ones — insert/update/delete/findOne, `findOneBypassCache` asserting cache hits still increase (140-147), nested-retry inside `executeTransactionalCommand` (72-111), and `testFactoryParameter` reading the configured factory parameter off the generated `PureOrderObjectFactory` (66-70); dated variants in `TestPureBitemporalTransactionalObject`, remote/three-tier in `test/pure/TestPureObjectsRemote.java`. `TestTempObject` covers context create/destroy with and without source attributes, inside/outside transactions, rollback dropping buffered inserts, and querying via `existsWithJoin`/relationship joins against temp tables (`TestTempObject.java:440-472,474-494`). `MithraObjectGraphExtractorTest` implements all three strategy interfaces itself (`test/util/MithraObjectGraphExtractorTest.java:54`) and asserts extracted-file contents per scenario (threshold exceeded, filters, ignored relationships, edge-point milestoning — 100-186); `DbExtractorTest` round-trips `testdata/DbExtractorTest_source.txt` through extract-then-compare (68-72). Reverse-engineering tests are thinner: `TableInfoTest` (FK pruning, class-name derivation) plus `DatabaseTableValidator/DatabaseIndexValidator/MaxLenValidator` tests in `reladomogenutil/src/test/.../generator/`.
