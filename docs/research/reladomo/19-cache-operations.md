## The per-class admin surface

`MithraRuntimeCacheController` (`mithra/util/MithraRuntimeCacheController.java`) wraps one finder class and is the unit of runtime cache administration: `clearQueryCache()` (line 104, portal query cache only), `reloadCache()` (116, full re-read from DB), `clearPartialCacheOrReloadFullCache()` (124 — clears if partial, reloads if full), `getCacheSize()` (161), `isPartialCache()` (156), and off-heap introspection `getOffHeapAllocated/UsedDataSize`, `...IndexSize` (391-409). `renewCacheForOperation(Operation)` (411) delegates to the portal and then clears the query cache; the implementation (`mithra/database/MithraAbstractDatabaseObject.java:2085-2140`) diffs the cached result of `findMany(op)` against a fresh DB read via a PK `FullUniqueIndex`, applies `cache.updateCache(new, updated, deleted)`, and returns a `RenewedCacheStats` (`mithra/util/RenewedCacheStats.java`) with the inserted/updated/deleted lists and elapsed time.

**Archive/export**: `archiveCache(OutputStream[, Filter])` (205-246) writes a two-byte header (`CACHE_ARCHIVE_VERSION = 2`, `CACHE_COMPRESSION_VERSION = 2`, lines 56-58) then an LZ4-compressed `ObjectOutputStream` containing class name, finder `serialVersionId`, and the serialized cache; `readCacheFromArchive` (248) replaces cache contents while `readCacheFromArchiveDoNotRemoveLeftOver` (253-261, optional date filter) merges.

Manager-wide operations on `MithraManager` (`mithra/MithraManager.java`): `clearAllQueryCaches()` (754) delegates to `MithraConfigurationManager.clearAllQueryCaches()` (`mithra/util/MithraConfigurationManager.java:1170-1186`), which snapshots the controller set and clears each — the javadoc notes the special per-transaction query cache is *not* cleared. Also `getRuntimeCacheControllerSet()` (759), `cleanUpRuntimeCacheControllers()` (670), `cleanUpPrimaryKeyGenerators()` (422 — clears `MithraPrimaryKeyGenerator` state), and `loadMithraCache(portals, threads)` (735).

## Startup full-cache population

`MithraConfigurationManager.initializeRuntime()` (267-334): for each connection-manager block, classes with `cacheType="full"` (or DB-replicated) are eagerly instantiated, and those with `loadCacheOnStartup` (XSD default `true`, `reladomo/src/main/xsd/mithraruntime.xsd:114`) are loaded in parallel by `loadMithraCache` with `initialLoaderThreads` (default 1) via a `ThreadConservingExecutor` (267-276). The per-portal load calls `MithraAbstractDatabaseObject.loadFullCache()` (2069-2078), whose operations come from a `LoadOperationProvider` (field default `DefaultLoadOperationProvider`, line 155). The default provider (`mithra/database/DefaultLoadOperationProvider.java:31`) returns `finder.all()`-style ops — `equalsEdgePoint()` on each as-of attribute for dated classes — and **throws** for classes with a source attribute, which must configure a custom `loadOperationProvider` on the connection manager or per object (`MithraConfigurationManager.java:910-916`). `reloadFullCache()` is the same code path (2080-2083). There is no built-in refresh scheduler; `reloadCache`/`runRefresh` are caller-triggered.

## The declarative cacheloader framework

`mithra/cacheloader/` loads/refreshes many full caches from XML (`reladomo/src/main/xsd/mithracacheloader.xsd`): a `<CacheLoader>` root (attributes `threadsPerDbServer` default 10, `syslogCheckThreshold`, `reportedSlowSQLTime`, `captureLoadingTaskDetails`) holds `<TopLevelLoader classToLoad=... sourceAttributes=... prerequisiteClasses=... factoryClass=...>` and `<DependentLoader relationship="Owner.relName" helperFactoryClass=...>` elements with `<Param>` children. `CacheLoaderManager` (`mithra/cacheloader/CacheLoaderManager.java:36-72`) exposes three loads plus one helper, each creating a `CacheLoaderContext` (`CacheLoaderManagerImpl.java`):

- `runInitialLoad(businessDates, initialLoadEndTime, monitor)` (38) — full load, then stamps `portal.setLatestRefreshTime(initialLoadEndTime)` on every configured class (83-94).
- `runRefresh(businessDates, refreshInterval, monitor)` (46) — loads only rows changed inside `RefreshInterval` (a start/end `Timestamp` pair, `RefreshInterval.java:26`), then stamps `refreshInterval.getEnd()`.
- `runQualifiedLoad(businessDates, classesToLoadWithAdditionalOperations, loadDependent, monitor)` (54) — subset load via `QualifiedByOperationLoadContext`; `loadDependentObjectsFor(ownerObjects, ...)` (155) uses `QualifiedByOwnerObjectListLoadContext`; the default is `NonQualifiedLoadContext` (everything qualifies).

`CacheLoaderContext.execute()` (`CacheLoaderContext.java:66-115`) drives the pipeline: each `TopLevelLoaderFactory` creates load tasks per business date per source attribute, tasks attach prerequisites, `DependentLoaderFactory`s create their spawners and attach to owner lists, then `CacheLoaderEngine.waitUntilAllTasksCompleted()` runs everything. Milestoning determines the factory: `FullyMilestonedTopLevelLoaderFactory` (business + processing `equalsEdgePoint`, `FullyMilestonedTopLevelLoaderFactory.java:37-38`), `BusinessDateMilestonedTopLevelLoaderFactory`, `ProcessingDateMilestonedTopLevelLoaderFactory`, `NonMilestonedTopLevelLoaderFactory`. Business dates are grouped into `DateCluster`s — dates within `MAX_DISTANCE = 5` days share one SQL range scan (`DateClusterCreator.java:31-39`); `BusinessDateMilestonedTopLevelLoaderFactory.getBusinessDateOp` (63-70) turns a multi-date cluster into `FROM_Z <= end AND THRU_Z > start`.

Refresh reuses the same tasks with extra operations: `AbstractLoaderFactory.createLoadTasks` branches when `context.getRefreshInterval() != null` (`AbstractLoaderFactory.java:90-92, 158-180`) and appends per-task `TaskOperationDefinition`s from the abstract `buildRefreshTaskDefinitions` (270) — e.g. the fully-milestoned version (`FullyMilestonedTopLevelLoaderFactory.java:50-60`) issues two ops, processing `IN_Z >= start` and `OUT_Z >= start`, so only rows milestoned since the last refresh are read.

Dependent loading is cursor-fed rather than query-per-owner: `DependentLoaderFactoryImpl` parses the `relationship` string into join attributes, filters owner objects out of the owner's cache with `CacheIndexBasedFilter` (`DependentLoaderFactoryImpl.java:70`), and funnels owner keys through a `DependentKeyIndex` into a `DependentLoadingTaskSpawner`, which batches keys into IN-clause loading tasks. The engine (`CacheLoaderEngine.java`) keeps one `LoadingTaskThreadPoolHolder` per database host/schema (`getOrCreateThreadPool`, 58-73; pool name e.g. `localhost:nystlew`) sized by `threadsPerDbServer`, using `IOTaskThreadPoolWithCpuTaskConveyor` and `DualCapacityBlockingQueue`. `CacheLoaderMonitor` (`CacheLoaderMonitor.java`) aggregates per-task `LoadingTaskMonitor`s, thread-pool monitors, and dependent-key-index monitors, and captures the first exception. `createCacheFilterOfDatesToKeep` (`CacheLoaderManagerImpl.java:120-152`) emits per-class `BooleanFilter`s matching the configured dates, usable with the archive/eviction APIs.

A minimal config (from `CacheLoaderManagerTest.java:87-91`):

```xml
<CacheLoader>
    <TopLevelLoader classToLoad="com.gs.fw.common.mithra.test.glew.LewContract" sourceAttributes="NYK"/>
    <DependentLoader relationship="com.gs.fw.common.mithra.test.glew.LewContract.lewTransaction"/>
</CacheLoader>
```

## Master-cache replication for full-cache clients

`<MasterCacheReplicationServer masterCacheId=... className=... syncThreads="5" syncIntervalInMilliseconds="60000">` in the runtime XML (`mithraruntime.xsd:87-96`) turns a JVM into a **replica** of another JVM's off-heap caches instead of loading from the DB. `MithraConfigurationManager.lazyInitReplicatedObjects` (488-544) instantiates the transport factory (any class with `static getInstance(Properties)` returning a `MasterCacheService`, 637-676), asks the master for its `RemoteMithraObjectConfig[]`, and forces every replicated class to `fullCache = true; offHeapFullCache = true` (520-521). `initializeRuntime` then calls `uplink.startSyncAndWaitForInitialSync(portals)` (298-310).

The protocol (`mithra/cache/offheap/MasterCacheService.java:22-31`) has four calls: `getObjectConfigurations()`, `retrieveInitialSyncSize()`, `retrieveStrings(startIndex)`, and `syncWithMasterCache(className, maxReplicatedPageVersion)`. Sync is **page-based**: the client reports the highest off-heap page version it has; the master's `FastUnsafeOffHeapDataStorage.sendSyncResult` (869) / `serializeSyncResult` (1224-1319) streams only pages whose version is newer, in batches (`sendPagesInBatches`, 1320); the client's `MasterSyncResult` (`MasterSyncResult.java:73-100`) reads them into `FastUnsafeOffHeapPageBuffer`s keyed by a master-page-index → local-location map. `MasterCacheUplink` (`mithra/cache/offheap/MasterCacheUplink.java`) sorts portals largest-first from the initial-size result (170-194), runs `syncThreads` `MasterCacheSynThread`s that repeatedly claim portals and call `portal.syncWithMasterCache(uplink)` (245-279), re-syncs every `syncInterval`, and supports `pause()`/`unPause()` (79-84). The string pool is replicated separately: `syncStrings()` (119-148) maps master string refs to local `StringPool` off-heap addresses. The server side is just `MasterCacheServiceImpl` (`MasterCacheServiceImpl.java:28-62`) exposed over any RPC transport.

## Testing patterns

`reladomo/src/test/java/com/gs/fw/common/mithra/test/cacheloader/` (suite `CacheLoaderSuite.java`): `CacheLoaderManagerTest` builds loader configs from inline XML strings, uses `MithraTestResource("MithraCacheTestConfig.xml")` with one H2 database per source attribute (`NYK`/`LN1`/`LN2`), asserts loaded objects via finders plus monitor introspection (task counts, pool name `localhost:nystlew`), and asserts the JVM runs in EST because refresh fixtures are timezone-sensitive (setUp, line 96). Refresh tests (`testRefresh*`, 313-475) insert milestoned rows then verify interval loads. Mocks: `MockAdditionalOperationBuilder`, `MockCacheLoaderEngine`, `MockTaskSpawner`, `MithraCacheLoaderRuntimeConfigTestUtil`. Master-cache replication is tested multi-VM: `CacheReplicationTestCase` extends `multivm/MultiVmTestCase`; the worker VM hosts `MasterCacheServiceImpl` behind a Jetty `PspServlet` (init-param `serviceClass.MasterCacheService`, line 157) with runtime `MithraConfigMasterCache.xml` (`offHeapFullCache="true"`), while the main VM reads `MithraConfigReplicantCache.xml`, whose `<MasterCacheReplicationServer ... className="...PspBasedMithraMasterServerFactory" syncIntervalInMilliseconds="2000"/>` (line 19) points at the worker.
