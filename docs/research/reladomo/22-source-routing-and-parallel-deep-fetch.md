## Source-attribute (shard) routing semantics

**Declaration.** `<SourceAttribute name="..." javaType="..."/>` is an optional, at-most-one element of a MithraObject (`reladomogen/src/main/xsd/mithraobject.xsd:121-126`). The XSD states it "does not correspond to a column in the database" and that "the only valid types for source attributes are int and String" (`mithraobject.xsd:248-251`). At runtime it is an ordinary `SingleColumnIntegerAttribute`/`SingleColumnStringAttribute` whose `columnName` is null — that *is* the discriminator: `isSourceAttribute()` returns `this.columnName == null` (`mithra/attribute/SingleColumnIntegerAttribute.java:105-108`, `SingleColumnStringAttribute.java:89-92`).

**Finder surface.** The generated finder emits the source attribute alongside normal attributes plus `getSourceAttribute()`/`getSourceAttributeType()` (`reladomogen/src/main/templates/readonly/Finder.jsp:82-84, 905-913`), so users write e.g. `TinyBalanceFinder.acmapCode().eq("A")`. A source restriction in the operation is **mandatory for any query that reaches SQL**: `eq()` on a source attribute registers itself as the query's `SourceOperation` (`mithra/finder/AtomicEqualityOperation.java:182-189`); if none is found when building SQL, `SqlQuery.getSourceAttributeValue()` throws `MithraBusinessException("could not find source attribute for <Finder> in operation ...")` (`mithra/finder/SqlQuery.java:561-569`). Only equality and IN are legal: range ops throw `"cannot do range operations on source attributes"` (`mithra/finder/GreaterThanEqualsOperation.java:42`).

**Cross-shard fan-out.** `in()` over a set of source values *is* the cross-shard query mechanism. Every set-based operation implements `SourceOperation`; `getSourceAttributeValueCount()` returns the set size (`mithra/finder/integer/IntegerInOperation.java:142-145`), which becomes `SqlQuery.getNumberOfSources()` (`mithra/finder/SqlQuery.java:626-631`; 1 for `eq`, `AtomicEqualityOperation.java:151-154`). `findSequential` then loops `for (sourceNum = 0; sourceNum < sourceCount; ...)`: per source it resolves connection/DatabaseType/timezone generically, calls `query.prepareQueryForSource(sourceNum, dt, timeZone)` (which may itself split into several chunked-union queries), executes the same SQL on each shard's connection, and appends all rows into **one** `MithraFastList` (`mithra/database/MithraAbstractDatabaseObject.java:1242-1341`); `processResults` applies the order-by across the combined list in memory (`:1343-1348`). An IN over sources is only legal on the *selected* object — using it behind a join throws `"cannot have multiple in operations for source keys"` (`IntegerInOperation.java:92-101`).

Worked example — `TinyBalanceFinder.findMany(op.and(acmapCode().in({"A","B"})))`:

```text
source "A": SELECT ... FROM TINY_BALANCE WHERE <pred>   (connection from connectionManager.getConnection("A"))
source "B": SELECT ... FROM TINY_BALANCE WHERE <pred>   (connection from connectionManager.getConnection("B"))
            → rows appended to one list, one order-by sort, one CachedQuery
```

**Connection mapping.** The generated database object forwards `source` to a user-supplied connection manager: `IntSourceConnectionManager`/`ObjectSourceConnectionManager` declare `getConnection(source)`, `getDatabaseType(source)`, `getDatabaseTimeZone(source)`, `getDatabaseIdentifier(source)`, `createBulkLoader(source)` (`mithra/connectionmanager/IntSourceConnectionManager.java:28-36`, `ObjectSourceConnectionManager.java:28-36`) — the shard→database/schema mapping is entirely the application's function.

**Cache segregation.** Loaded rows are stamped with the source they came from: `processResultSet` calls `inflateDataGenericSource(rs, source, dt)` so the columnless attribute gets a value (`mithra/database/MithraAbstractDatabaseObject.java:1580-1588`). The generated `Finder.getPrimaryKeyAttributes()` appends the source attribute after the real PK (`readonly/Finder.jsp:1236-1247`), and those extractors seed every cache index (`generator/MithraObjectTypeWrapper.java:3714-3728`, both dated and non-dated) — so identical PKs from different shards are distinct cache entries, and the generated PK `eq(other)` operation includes the source equality (`readonly/Finder.jsp:984-993`). `CacheRefresher` ANDs `sourceAttribute.nonPrimitiveEq(value)` into refresh ops and segregates dirty objects per source before refreshing (`mithra/cache/CacheRefresher.java:49-56, 111-160`). Full-cache preload of a source-attributed class has no default: `DefaultLoadOperationProvider` throws, demanding a custom `loadOperationProvider` (`mithra/database/DefaultLoadOperationProvider.java:35`). Class-level notification subscription requires an explicit source-value set (`mithra/portal/MithraAbstractObjectPortal.java:1375-1395`).

**Writes.** DML never needs a source *operation*: the routing value is read off the object's data. `zInsert` derives `source = getSourceAttributeValueFromObjectGeneric(dataToInsert)` and takes `getConnectionForWriteGenericSource(source)` (`mithra/database/MithraAbstractDatabaseObject.java:3580-3593`); `zUpdate`, `zDelete`, batch/multi variants do the same from the first object's data (`:3646-3654, 3761, 3827`), and tuple inserts segregate a mixed list by source first (`:2796-2811`). Refresh likewise (`:2226-2229`).

## Multi-threaded deep fetch and cross-database adhoc deep fetch

**Where parallelism enters.** `MithraList.setNumberOfParallelThreads(n)` (`mithra/MithraList.java:106`) is stored on the operation-based delegate (`mithra/list/AbstractOperationBasedList.java:143-152`) and used twice at resolve time: (1) the **root query** is passed `n` through `findAsCachedQuery` (`:220-221`); when `n > 1` and no row limit, the database layer switches from `findSequential` to `findParallel` (`mithra/database/MithraAbstractDatabaseObject.java:1233-1240`), which runs **one task per source and per chunked-union query within a source** on an `AutoShutdownThreadExecutor` of `n` threads, each task producing a per-thread result list merged via a blocking queue; a `CountingLock` keeps statement preparation in query order (`:1351-1470`). (2) **Deep fetch**: `resolveDeepRelationships` forwards `n` to `DeepFetchNode.deepFetch` (`AbstractOperationBasedList.java:355-362`).

**What deep fetch parallelizes.** `DeepFetchNode.deepFetch` builds a `ThreadConservingExecutor(n)` — or a run-on-caller-thread executor when `n == 1` — and forces `n = 1` inside a transaction (`mithra/finder/DeepFetchNode.java:230-247`). The unit of work is **one `DeepFetchRunnable` per relationship node** in the deep-fetch tree (`:319-323, 1060-1105`): sibling relationships fetch concurrently; a child node's runnable is only spawned by its parent's runnable after the parent resolves, and completion propagates by child-count decrement (`:1106-1129`). The executor uses `n-1` daemon threads plus the calling thread inside `finish()` (`mithra/util/ThreadConservingExecutor.java:72-82`). Per-source fan-out *within* each relationship query stays sequential (it goes through `findSequential` unless that node's own list is parallel). `MithraCompositeList`/`MithraCpuBoundThreadPool` are a separate mechanism — parallel *in-memory cache matching* for large lists (e.g. `OrOperation.applyOperation`, `mithra/finder/OrOperation.java:276-341`) — not deep fetch.

**Adhoc deep fetch.** A non-operation list (`AdhocFastList`) has no root operation, so `deepFetchAdhocList` starts from the concrete objects (`DeepFetchNode.java:455-462`). The pipeline (`deepFetchChildrenForAdhoc`, `:464-508`):

1. **In-memory first**: each child tries `deepFetchFirstLinkInMemory` — per parent, `mapper.getOperationFromOriginal(parent)` is probed against the query cache/full cache, partitioning parents into found/not-found (`mithra/finder/SimpleToOneDeepFetchStrategy.java:77-141`). If ≥ 80% resolved (`PERCENT_COMPLETE_TO_IGNORE`, `DeepFetchNode.java:88`) and the strategy `canFinishAdhocDeepFetchResult()`, only the not-found remainder goes to the database (`finishAdhocDeepFetch`, `SimpleToOneDeepFetchStrategy.java:150-172`).
2. **Segregate by source**: the adhoc list is split into per-shard sublists via the source attribute (`segregateBySource`, `DeepFetchNode.java:751-784`), and steps below run once per shard — an adhoc list mixing sources issues per-source queries (`deepFetchManyChildrenAdhoc`, `:535-637`).
3. **Batch by join shape**: children sharing the same left-attribute set, source extractor, and `PersisterId` are grouped under one `DeepFetchKey`; each group is fetched with a single-attribute `IN` clause (< `MAX_SIMPLIFIED_IN` = 1000, `mithra/finder/DeepRelationshipUtility.java:40-41`) or one shared temp-table join (`deepFetchWithTempContext`/`createTempTableForDeepFetch`, `:680-706, 862`).

**Cross-database case.** `getSimplifiedJoinOp` checks `differentPersisterIdThanParent(mapper)` — it collects all portals the mapper touches and compares `PersisterId`s, which are allocated one per connection-manager instance (`DeepFetchNode.java:930-955`; `mithra/util/MithraConfigurationManager.java:201-209`). When parent and child live in different databases, `maxSimplifiedIn` becomes `Integer.MAX_VALUE`: the child query is always expressed as `rightAttribute.in(<parent values>)` against the child's own database (`mithra/finder/EqualityMapper.java:577-582`) — no SQL join or temp table spanning connections is ever attempted, regardless of parent-list size.
