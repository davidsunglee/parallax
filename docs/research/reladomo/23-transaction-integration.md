# Nesting joins the outer transaction (no savepoints); JTA/XA integration spans container TMs, pooled XA connections, and JMS two-phase commit

> repository (`../reladomo`). Path abbreviations: **`mithra/`** =
> `reladomo/src/main/java/com/gs/fw/common/mithra/`; **`xa/`** =
> `reladomoxa/src/main/java/com/gs/reladomo/jms/`.

**Nesting is a join, not a savepoint.** `MithraManager.startOrContinueTransaction(style)` (`mithra/MithraManager.java:246-282`) checks the thread-local: if a transaction exists it just wraps it in `new MithraNestedTransaction(parent)` (252) — no new JTA transaction, no savepoint, and the `TransactionStyle` (timeout/retries) is ignored for the nested case. `MithraNestedTransaction` (`mithra/transaction/MithraNestedTransaction.java:38`) holds only a `parent` pointer; every operational method (`insert`, `update`, `enrollObject`, `executeBufferedOperations`, `getTimeoutInMilliseconds`, …) walks up via `getMithraRootTransaction()` (86-92) and delegates, so timeout and all buffered state are strictly the root's. `startedTransaction()` returns `false` and `setStarted` is a no-op (276-285).

- `nested.commit()` (57-68) commits **nothing** — it optionally flushes buffered SQL when `setFlushNestedCommit(true)` was called (346-349), then pops the thread-local back to the parent (`MithraManager.popTransaction`, `MithraManager.java:297-307`). The javadoc states it directly: "Calling commit on a nested transaction does nothing. Calling rollback on a nested transaction rollsback the entire transaction" (`MithraManager.java:228-234`).
- `nested.rollback()` (71-75) calls `root.setExpectRollback()` and pops; `rollbackWithCause` (78-84) also stores the cause on the root. The **outer** `MithraRootTransaction.commit()` then refuses: "cannot commit a transaction that expects a rollback (most probably a nested transaction failed" (`mithra/transaction/MithraRootTransaction.java:820-825`), thrown as a `MithraTransactionException` whose cause and retriability are copied from the nested failure (`throwExceptionWithCause`, 899-910).

**Ambient behavior of `executeTransactionalCommand`.** Inside an existing transaction it does not even create a nested wrapper: it runs `command.executeTransaction(currentTx)` directly, and on a non-`RuntimeException` throwable marks the tx `expectRollbackWithCause` — **no retries** in the inner frame (`MithraManager.java:524-545`). Retry bookkeeping honors nesting too: `MithraTransaction.handleTransactionException` (`mithra/MithraTransaction.java:125-192`) only counts an exception as retriable when `tx.startedTransaction()` is true, logging "failed in inner transaction" otherwise (162-169) and rethrowing so the *outer* retry loop (which did start the tx) performs the actual retry.

**Joining a container-managed JTA tx.** Besides `setJtaTransactionManagerProvider(JtaProvider)` (`MithraManager.java:223-226`), `joinJtaTransaction(jtaTx)` (352-387) lets Reladomo enlist in a transaction it did not begin: it requires an ACTIVE status, builds a `MithraRootTransaction` with `setStarted(false)` (so Reladomo will flush but never call TM `commit()` — see `MithraRootTransaction.commit()` 836-848, which just `removeTransaction`s when `!startedTransaction`), and registers it as a JTA `Synchronization` (`createMithraRootTransaction`, 404-410). `leaveJtaTransaction()` (389-402) must be called at the same nesting level (throws if current tx is nested).

**XAConnectionManager** (`mithra/connectionmanager/XAConnectionManager.java:33`) is an `AbstractConnectionManager` whose pool hands out transaction-affine JDBC connections via `XAConnectionPoolingDataSource` (`mithra/connectionmanager/XAConnectionPoolingDataSource.java:45`): `getConnection()` (111) checks the current Mithra transaction and, per transaction, creates one `JDBCConnectionXAResource` (an `XAResource` + `LocalTm.SinglePhaseResource`, 359) enlisted via `transaction.enlistResource(resource)` (81). Its `commit(Xid)`/`rollback(Xid)` (545/490) commit or roll back the raw JDBC connections held for that tx — i.e. JDBC local transactions adapted into the XA protocol.

**reladomoxa: exactly-once JMS + DB.** The module (package `com.gs.reladomo.jms`) coordinates JMS XA sessions and the database in one transaction managed by `MultiThreadedTm` (`mithra/transaction/MultiThreadedTm.java`), a bundled `TransactionManager` that rejects JTA-level nesting (`begin()` at 32 throws `NotSupportedException` if a tx exists) whose `MultiThreadedTx` performs a real two-phase commit: `prepare()` per branch, then `commit` (`mithra/transaction/MultiThreadedTx.java:305-321, 245-293, 469-491`), with special-casing for `LocalTm.SinglePhaseResource` (823) and a one-phase-commit decision for single-branch txs (940-945).

- `OutgoingAsyncTopic` (`xa/OutgoingAsyncTopic.java:37`) sends message batches on a dedicated thread; `asyncSendMessages` (65) enlists a `FutureXaResource` wrapping the JMS `XASession`'s `XAResource` (77, 260) and returns a `Future` so DB work overlaps the send. `IncomingTopic.enlistIntoTransaction()` enlists the consumer's `XAResource`, throwing `RestartTopicException` on failure (`xa/IncomingTopic.java:49-62`).
- `BatchJmsMessageLoop` (`xa/BatchJmsMessageLoop.java:45`) is the driver: per iteration it starts a Mithra tx, fills a batch from the incoming topic, calls the user's `BatchProcessor`, and commits (193-206); `MithraBusinessException`, `RollbackException`, and `RestartTopicException` each get dedicated recovery paths (212-226).
- Crash recovery uses an **xid record stored via Reladomo itself**: `TopicResourcesWithTransactionXid.startTransaction()` (`xa/TopicResourcesWithTransactionXid.java:129-134`) writes the global transaction id with `committed=false` into a `ReladomoTxIdInterface` object (`mithra/../gs/reladomo/txid/ReladomoTxIdInterface.java`) in a *separate* transaction (136-150); `commit()` flips `committed=true` inside the main tx, flushes, waits for outgoing-send futures, then commits (162-168). After a topic failure, `restartTopics` re-connects and calls `XAResource.recover(TMSTARTRSCAN|TMENDRSCAN)` (258-260), using the persisted `committed` flag to decide whether in-doubt branches are committed or rolled back.

## Testing patterns

Nested semantics: `TestTransactionalObject.testNestedRollbackOfInnerMethodWhenOuterMethodFails` / `...OfOuterMethodWhenInnerMethodFails` / `testNestedRollbackBecauseOfDbTimeout` (`reladomo/src/test/java/com/gs/fw/common/mithra/test/TestTransactionalObject.java:1266,1315,1390`) invoke generated `TransactionalMethodSignature` wrappers that nest transactions, then verify via raw JDBC that *both* frames rolled back. `MithraTestResource` installs `new DefaultJtaProvider(new LocalTm())` (`reladomo/src/test-util/java/com/gs/fw/common/mithra/test/MithraTestResource.java:540`). The XA/JMS side runs against a fully in-memory XA-capable broker (`reladomoxa/src/test/java/com/gs/reladomo/jms/InMemoryBroker.java`, `InMemoryXaResource.java` etc.): `BatchJmsMessageLoopTest.setUp` installs an anonymous `JtaProvider` returning a fresh `MultiThreadedTm` (`BatchJmsMessageLoopTest.java:70-79`), and tests inject messages, wait for batch commits, and assert echo-topic output and rollback flags (92-114); `OutgoingAsyncTopicTest` covers the send path.
